# Team Protocols

仅有 Agent 间的通信能力还不够。缺少明确的消息契约和状态约束时，请求可能被重复处理、错误响应可能关联到其他任务，Agent 也无法可靠判断一次协作是否已经完成。

Team Protocols 通过“请求—响应—确认”机制和 requestId，将 Agent 协作过程协议化：

请求方创建唯一的 requestId，记录协议类型、参与方、请求内容和当前状态，然后发送请求。

接收方根据消息类型将请求路由到对应的处理器，并返回携带同一 requestId 的响应。

请求方收到响应后，校验 requestId、协议类型、发送方及当前状态，更新请求状态并关闭本次协议。

每个请求由状态机管理，例如：

pending → approved | rejected | timed_out | cancelled

对于重复响应、未知请求、类型不匹配和已经结束的请求，系统应拒绝或幂等忽略。同时还需要定义超时、重试、取消和状态持久化机制，保证 Agent 或进程重启后仍能恢复未完成的协作。

消息路由负责回答“这是什么消息，应交给哪个处理器”；协议状态机负责回答“这条请求进行到了哪一步，下一步操作是否合法”。对于审批、权限申请和关机等协议，只有状态满足条件后才能执行相应操作，从而让整个协作过程可追踪、可验证、可恢复。
